# Fixing DNS Resolution Issues with Tailscale on Raspberry Pi

*Resolving DNS conflicts when Tailscale VPN manages /etc/resolv.conf*

---

## Problem Overview

### Symptom

Package installation fails with DNS resolution errors:

```
Temporary failure resolving 'deb.debian.org'
Error: Unable to fetch some archives
```

Commands like `apt update` and `apt install` fail because the system cannot resolve domain names.

### Root Cause

Tailscale VPN takes control of `/etc/resolv.conf` and sets its own DNS server (`100.100.100.100`) as the primary nameserver. While this works for Tailscale's internal domains (`*.ts.net`), it may fail to resolve external public domains depending on network configuration.

The file contains a warning:
```
# DO NOT EDIT THIS FILE BY HAND -- CHANGES WILL BE OVERWRITTEN
```

Any manual edits to add public DNS servers are immediately reverted by Tailscale's management daemon.

---

## Diagnostic Steps

### Step 1: Verify Internet Connectivity

```bash
ping -c 3 8.8.8.8
```

Tests if the Pi can reach the internet by IP (bypassing DNS). If this fails, the problem is routing or internet connectivity — not DNS.

### Step 2: Test DNS Resolution

```bash
ping -c 2 deb.debian.org
```

If you see `Temporary failure in name resolution`, DNS is broken while internet connectivity is fine.

### Step 3: Inspect DNS Configuration

```bash
cat /etc/resolv.conf
```

If you see:
```
nameserver 100.100.100.100
search tail*.ts.net
```

Tailscale is managing DNS and its resolver is failing for public domains.

---

## Solution

### Step 1: Disable Tailscale DNS Management

```bash
sudo tailscale set --accept-dns=false
```

This tells Tailscale to stop modifying `/etc/resolv.conf`. Persists across reboots — run once.

### Step 2: Set Public DNS Servers

```bash
echo -e "nameserver 8.8.8.8\nnameserver 1.1.1.1" | sudo tee /etc/resolv.conf
```

- `8.8.8.8` — Google Public DNS
- `1.1.1.1` — Cloudflare DNS (fallback)

### Step 3: Verify

```bash
ping -c 2 deb.debian.org
```

Expected:
```
PING debian.map.fastlydns.net (146.75.126.132) 56(84) bytes of data.
64 bytes from 146.75.126.132: icmp_seq=1 ttl=56 time=13.8 ms
```

---

## Why Not Just Edit resolv.conf Directly?

Tailscale immediately overwrites any manual changes. The only reliable fix is disabling Tailscale's DNS management first, then setting your preferred DNS servers.

Configuring DNS in `dhcpcd.conf` also doesn't help — Tailscale still takes priority and overwrites the file.

---

## Impact on Tailscale Functionality

Disabling DNS management does **not** break Tailscale VPN.

**What still works:**
- SSH via Tailscale IP address (e.g., `ssh user@100.96.121.27`)
- All Tailscale encrypted tunneling and routing
- Access to other Tailscale devices by IP

**What stops working:**
- MagicDNS — resolving device hostnames like `device-name.ts.net` automatically
- SSH via short hostname (e.g., `ssh device-name`)

**Workaround for MagicDNS:** Add entries to `/etc/hosts`:

```bash
sudo nano /etc/hosts
```

Add a line like:
```
100.96.121.27   your-device-name
```

This allows `ssh your-device-name` to work without MagicDNS.

---

## Prevention

On new Pi installations with Tailscale:

```bash
# 1. Install Tailscale, then immediately:
sudo tailscale set --accept-dns=false

# 2. Set public DNS:
echo -e "nameserver 8.8.8.8\nnameserver 1.1.1.1" | sudo tee /etc/resolv.conf

# 3. Verify:
ping -c 2 deb.debian.org
```

Do this before running `apt install` for anything else.
